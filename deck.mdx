# What Are Bloom Filters?

> _In-memory probablistic data structure that can say if a key either does **NOT** exist in a set, or that it **MIGHT** exist in the set_

</Preso>

---

<Preso>

# What Do They Do?
* **Protect** more expensive resources
* Databases, files, or other services

</Preso>

---

<Preso>

# How to Use?
* Construct a filter for a set in the more expensive resource
* Insert each key from this target set into the filter
* When performing a lookup, check filter before the target set

</Preso>

---

<Preso>

# Remember Hashing?
* Given key `k`, calculate `h` via `h(k)`
* Use `h` to find a value in a table of size `m`
* Looking up a key performs the same `h(k)` to check for the presence of the key

</Preso>

---


<Preso>

# Bloom Filters Use Multiple Hashes
* Table is a bit array
* Calculate different hash values fot each key `k`
* Mark each bit in the array at the hash value to `1`

</Preso>

---

<Preso>

# Inserting a Key `k`
* `m` size of bit array; `i` number of hashes
* Calculate `i` hashes for `k`
* For each `h`, find its position within the bit array and set it to 1

</Preso>

---

<Preso>

# Might Contain a Key `k`
* Calculate the **same** hash values
* For each `h`, find the bit position
* Logical `&` of the bits at each position
* If `true`, then `k` **might** exist in the set

</Preso>

---

<Preso>

# Probablistic
* As keys are inserted, bit array becomes full (many bits set to 1)
* False positives will start to occur
* To reduce false positives, combination of growing table and increasing `i`

</Preso>

# Other Bloom Filter Properties

---

<Preso>

# Theoretical Performance?
* Insert and lookup is `O(i)`
* Stays constant as number of keys grow
* Choose faster hashes and over hashes with fewer collisions

</Preso>

---

<Preso>

# How Big are They?
* Only need a small amount of memory relative to the size of the set that built it
* Blacklist of 3 million entries ~177 MB; Bloom Filter is ~2.3 MB for with 95% false positive

</Preso>

---

<Preso>

# Bloom Filters are Constrainable
* You can have a _really_ small bloom filter if you want
* Simply pay the cost in increased false positives
* Do not grow linearly with respect to expected number of keys

</Preso>


---

<Preso>

# Downside?
* Can't delete!
* Setting a bit back to 0 from 1 affects multiple keys
* Must rebuild the filter

</Preso>

---

# So... What About this Offheap Part?

---

<Preso>

# Must Stay In Memory
* Affects Garbage Collections
* Systems like Roger can't afford a major GC
  * At least in its current form

</Preso>

---

<Preso>

# Offheap Memory Doesn't Get GC'ed
* Manual memory management
* But can be tricky with complex objects

</Preso>

---

<Preso>

# Bloom Fiters are Easy in Offheap Memory
* Bit array as a byte array using bit masks is reasonably easy
* Calculate expected size and ask for it
* Slower than the heap, but still faster than checking a data store

</Preso>

---

<Preso>

# Other Things to Track
* Hashing functions are still on-heap - could potentially use static functions
* Hash count also needs to be stored - could just but stored at the beggining of the filter array

</Preso>
